---
title: "Quickstart"
description: "Initialize a Mailman project and make your first HTTP request in minutes."
---

This guide walks you through initializing a Mailman project, creating request files, and running your first request.

<Steps>
  <Step title="Initialize a project">
    Run `mailman init` in your project directory:

    ```bash
    mailman init
    ```

    Mailman creates a `config.json` and a `requests/` directory with sample request files:

    ```
    Created config.json
    Created requests/ directory
    Created requests/get-users.json
    Created requests/create-user.json

    Mailman initialized successfully!
    Run 'mailman list' to see available requests
    ```

    <Note>
      If `config.json` already exists, Mailman skips it. Existing request files are also preserved.
    </Note>
  </Step>

  <Step title="Review the generated files">
    Open `config.json` to see your environment configuration:

    ```json config.json
    {
      "defaultEnvironment": "dev",
      "environments": {
        "dev": {
          "variables": {
            "baseUrl": "http://localhost:3000",
            "exampleVariable": "dev-example-variable"
          }
        },
        "prod": {
          "variables": {
            "baseUrl": "https://api.example.com",
            "exampleVariable": "prod-example-variable"
          }
        }
      }
    }
    ```

    And look at the sample request file in `requests/get-users.json`:

    ```json requests/get-users.json
    {
      "name": "Get Users",
      "method": "GET",
      "url": "{{baseUrl}}/users",
      "headers": {
        "Authorization": "Bearer {{API_KEY}}"
      }
    }
    ```

    The `{{baseUrl}}` placeholder is automatically replaced with the matching environment variable when you run the request.
  </Step>

  <Step title="List available requests">
    Run `mailman list` to see all requests in the `requests/` directory:

    ```bash
    mailman list
    ```
  </Step>

  <Step title="Preview a request before running">
    Use `mailman view` to see the fully resolved request without sending it — useful for verifying variable interpolation:

    ```bash
    mailman view requests/get-users.json
    ```

    This shows the method, URL, headers, and body with all variables substituted from the default environment.
  </Step>

  <Step title="Run your first request">
    Execute the request against your default environment:

    ```bash
    mailman run requests/get-users.json
    ```

    Mailman prints the response status, timing, headers, and body:

    ```
    Running: Get Users
    Environment: dev
    GET http://localhost:3000/users

    Status: 200 OK
    Time: 42ms

    Response Headers:
      content-type: application/json

    Response Body:
    [
      { "id": 1, "name": "Alice" },
      { "id": 2, "name": "Bob" }
    ]
    ```

    <Note>
      If your request contains variables that cannot be resolved (like `{{API_KEY}}`), Mailman prompts you to enter them interactively before sending.
    </Note>
  </Step>

  <Step title="Run against a different environment">
    Use the `-e` flag to specify an environment:

    ```bash
    mailman run requests/get-users.json -e prod
    ```

    This uses `baseUrl` from the `prod` environment in `config.json`.
  </Step>

  <Step title="Override variables at runtime">
    Pass variable values directly on the command line with `-v`:

    ```bash
    mailman run requests/get-users.json -v API_KEY=mytoken -v limit=50
    ```

    Runtime `-v` arguments have the highest priority and override all other sources.
  </Step>
</Steps>
